# 🚀 Studyroom Reservation & Recommendation

Spring Boot 기반 스터디룸 예약 시스템을 JSP 환경에서 리팩토링하고, 학번 기반 로그인/권한 관리와 혼잡도 기반 추천 기능까지 확장한 프로젝트입니다.  
기존 예약 CRUD 흐름을 Spring MVC, JPA, Spring Security 구조로 재정비하면서 추천 API, 전역 예외 처리, 예약 동시성 제어, 권한별 UI 분기까지 단계적으로 고도화했습니다.

# 📋 프로젝트 개요

이 프로젝트는 스터디룸 예약 과정을 더 안정적이고 서비스답게 개선하는 데 초점을 맞췄습니다.  
단순히 예약을 저장하는 수준을 넘어, 로그인 사용자 기준 예약 조회 범위를 분리하고, 중복 예약을 동시성 관점에서 방어하며, 사용 목적과 혼잡도를 반영한 추천 기능까지 제공하도록 확장했습니다.

### 프로젝트 목표

- 예약 등록, 조회, 삭제 흐름을 Spring MVC 기반으로 명확히 분리
- 사용자와 관리자 권한에 따라 다른 데이터와 UI를 제공
- 같은 날짜와 시간대에 대한 중복 예약을 안정적으로 차단
- 예약 목적과 혼잡도를 함께 고려한 추천 결과 제공
- 실패 상황에서도 다시 입력하기 쉬운 화면 흐름 구성

### 구현 범위

- `JSP` 메인 화면에서 로그인, 예약 등록, 예약 목록 조회를 통합
- `Spring Security` 기반 학번 로그인과 `USER`/`ADMIN` 권한 관리 구현
- `JPA` 기반 예약 데이터 접근 계층 리팩토링
- `/api/recommend` 추천 API와 점수 계산 로직 구현
- DB 유니크 제약조건과 예외 처리 기반 중복 예약 방지
- 관리자 전용 검색/차트 UI 및 권한별 데이터 필터링 구현

### 주요 담당 업무

- 예약/추천/인증 흐름 전반에 대한 백엔드 구조 설계 및 구현
- JSP 기반 메인 화면, 로그인 UI, 예약 폼, 목록/차트 화면 통합
- 권한별 조회 분기와 Spring Security 태그 기반 UI 제어 구현
- 추천 알고리즘, 예외 처리, 입력값 복구 UX 개선
- DB 유니크 제약조건을 활용한 동시성 제어 적용

### 핵심 성과

- 인증, 권한, 추천, 예외 처리, 동시성 제어를 포함한 서비스형 예약 시스템으로 구조를 확장
- 사용자 예약 흐름과 관리자 운영 흐름을 하나의 화면에서 자연스럽게 연결
- 단순 CRUD를 넘어 실제 운영 상황에서 발생할 수 있는 중복 예약과 로그인 예외 흐름까지 대응

# 🏗 시스템 구조

```text
Browser (JSP UI)
   │
   ├─ 예약 등록 / 로그인 / 조회 / 삭제
   └─ 추천 요청(fetch /api/recommend)
   ▼
Spring MVC Controller
   │
   ├─ ReservationController
   └─ RecommendationApiController
   ▼
Service Layer
   │
   ├─ ReservationService
   ├─ RecommendationService
   └─ CustomUserDetailsService
   ▼
JPA Repository
   │
   ├─ ReservationRepository
   └─ MemberRepository
   ▼
MySQL
```

설명

- JSP 화면은 예약 등록 폼, 로그인 영역, 예약 목록, 관리자 전용 차트를 하나의 메인 화면에서 제공합니다.
- `ReservationController`는 페이지 렌더링과 등록/삭제 흐름을 담당하고, `RecommendationApiController`는 추천 결과를 JSON으로 반환합니다.
- 서비스 계층에서 예약 저장, 권한별 조회 분기, 추천 점수 계산, Spring Security 사용자 조회를 분리해 MVC 구조를 명확히 했습니다.
- 데이터 무결성은 JPA 엔티티와 DB 유니크 제약조건에서 함께 보장하고, 예외는 전역 핸들러에서 처리합니다.

# 🗄 데이터베이스 설계

### **엔티티 설계**

설명

- `Member`
- 학번(`studentId`)을 PK로 사용하고, 암호화된 비밀번호, 이름, 권한(`USER`, `ADMIN`)을 저장합니다.
- `Reservation`
- 예약 번호(`id`)를 PK로 사용하고, 예약자 이름, 학번, 날짜, 시간대, 인원, 목적을 저장합니다.
- 예약 데이터는 학번을 함께 저장해 로그인 사용자 기준 개인 예약 조회가 가능하도록 구성했습니다.

### **데이터 무결성**

설명

- `Reservation` 테이블에 `(date, time_slot)` 유니크 제약조건을 두어 동일 날짜/시간대 중복 예약을 DB 레벨에서 차단했습니다.
- 인원 수는 `@Min(2)`와 `@Max(8)`로 제한해 비정상 입력을 방지했습니다.
- 비밀번호는 `BCryptPasswordEncoder`로 암호화하여 저장합니다.

### **조회 성능 최적화**

설명

- 예약 목록은 `findAllByOrderByDateAscTimeSlotAsc`, `findByStudentIdOrderByDateAscTimeSlotAsc`처럼 정렬이 포함된 메서드로 바로 조회해 화면 로직을 단순화했습니다.
- 권한별 조회를 서비스 레이어에서 분기해 관리자와 일반 사용자의 데이터 범위를 명확히 분리했습니다.
- 추천 기능은 특정 날짜 데이터만 `findByDate`로 조회한 뒤 메모리에서 시간대별 집계를 수행해 계산 범위를 줄였습니다.

# ✨ 주요 기능

### 학번 기반 로그인과 권한별 예약 조회를 하나의 흐름으로 통합

- 학번 기반 자체 로그인과 `USER`/`ADMIN` 권한 관리를 구현했습니다.
- 일반 사용자는 본인 예약만 조회하고, 관리자는 전체 예약 조회/검색/차트 확인이 가능합니다.
- JSP에서 Spring Security 태그를 사용해 로그인 여부와 권한에 따라 입력창, 목록, 관리자 기능 노출을 제어했습니다.

### 혼잡도와 이용 목적을 함께 반영한 시간대 추천 API 제공

- 날짜와 목적을 입력하면 `/api/recommend`가 상위 2개의 추천 시간대를 JSON으로 반환합니다.
- 추천 점수는 시간대별 예약 건수에 따른 혼잡도 감점과, 공부/팀플 목적에 따른 시간대 적합도 감점을 합산해 계산합니다.
- 추천 결과에는 점수뿐 아니라 "왜 이 시간대를 추천하는지"에 대한 자연어 설명을 함께 제공합니다.

### 예약 실패 시에도 다시 입력하기 쉬운 예외 처리와 복구 UX 구성

- 예약 등록 시 클라이언트 검증, 서버 예외 처리, 입력값 복구를 함께 적용해 실패 경험을 줄였습니다.
- 중복 예약은 DB 제약조건과 예외 처리로 막고, 사용자는 메인 화면에서 즉시 오류 메시지와 이전 입력값을 다시 확인할 수 있습니다.
- 관리자는 차트로 날짜별 예약 추이와 시간대별 선호도를 한 번에 확인할 수 있습니다.

# 🔍 주요 트러블슈팅

## DB 유니크 제약조건으로 동시 요청 상황의 중복 예약을 차단

**문제**

애플리케이션 레벨에서만 중복 예약을 검사하면, 동시에 두 요청이 들어올 때 모두 통과해 같은 시간대 예약이 중복 저장될 수 있었습니다.

**해결**

- `Reservation` 엔티티에 `(date, time_slot)` 유니크 제약조건을 추가했습니다.
- `ReservationService`에서 저장 시 `DataIntegrityViolationException`을 받아 중복 예약 상황을 감지하도록 변경했습니다.
- `GlobalExceptionHandler`에서 중복 예약을 `409 Conflict` 성격의 메시지로 변환해 API 응답과 로그를 일관되게 관리했습니다.

**결과**

- 동시 요청 상황에서도 DB가 최종 방어선 역할을 하도록 구조를 개선했습니다.
- 중복 예약 방지 로직이 애플리케이션 체크에만 의존하지 않게 되어 안정성이 높아졌습니다.
- 사용자는 "이미 예약된 시간대"라는 명확한 피드백을 받아 다시 입력할 수 있게 되었습니다.

## 메인 화면 통합 로그인에서 무한 리다이렉트와 입력값 유실을 해결

**문제**

Spring Security 로그인 페이지를 메인 화면(`/`)에 통합하는 과정에서 JSP 포워드/에러 디스패치와 충돌해 무한 리다이렉트가 발생하고, 예약 실패 시 입력값이 사라지는 UX 문제가 있었습니다.

**해결**

- `SecurityConfig`에서 `FORWARD`, `ERROR`, `INCLUDE` 디스패치 타입을 인증 없이 통과시키도록 설정했습니다.
- 로그인 페이지와 로그인 처리 URL을 메인 화면 구조에 맞게 재구성했습니다.
- `RedirectAttributes`를 사용해 에러 메시지와 이전 입력값을 플래시 속성으로 전달하도록 변경했습니다.

**결과**

- 메인 화면 기반 통합 로그인 UI가 안정적으로 렌더링되도록 개선했습니다.
- 예약 실패 후에도 이름, 날짜, 시간대, 목적 등의 입력값이 유지되어 재입력 부담이 줄었습니다.
- 로그인/예약 흐름이 한 화면 안에서 자연스럽게 이어지는 UX를 만들 수 있었습니다.

# 🎨 UX 개선

- 로그인 폼을 메인 페이지 우측 상단에 통합해 예약 흐름을 끊지 않도록 구성했습니다.
- 비로그인 사용자는 예약 입력 UI를 보더라도 실제 제출이 불가능하도록 비활성/안내 메시지를 제공했습니다.
- 클라이언트 검증 스크립트로 이름, 학번, 날짜, 시간대, 인원, 목적을 즉시 검사해 서버 왕복을 줄였습니다.
- 추천 결과를 로딩 상태, 결과 없음, 추천 Top 2 형태로 나눠 보여주어 사용자가 다음 행동을 쉽게 선택할 수 있게 했습니다.
- 관리자 전용 검색과 차트는 권한이 있는 사용자에게만 노출되도록 설계했습니다.

# 📈 회고 및 성장

### Learning

- Spring MVC에서 Controller, Service, Repository 책임을 분리하면 화면 로직과 비즈니스 로직을 훨씬 안정적으로 관리할 수 있다는 점을 배웠습니다.
- 동시성 문제는 애플리케이션 체크만으로 충분하지 않으며, DB 제약조건이 가장 확실한 방어선이라는 점을 경험했습니다.
- Spring Security는 인증 자체보다도 로그인 페이지 경로, 디스패치 타입, JSP 태그 연동까지 함께 설계해야 실제 서비스 UX가 안정된다는 점을 익혔습니다.

### Improvement

- 현재 날짜와 시간대를 문자열로 다루고 있어, 이후에는 `LocalDate`와 별도 시간대 타입으로 바꾸면 타입 안정성이 더 좋아질 수 있습니다.
- 테스트 코드가 부족하므로 추천 알고리즘, 권한별 조회, 중복 예약 처리에 대한 단위/통합 테스트 보강이 필요합니다.
- 예약 취소 권한, 관리자 전용 삭제 제약, 예외 흐름의 컨트롤러 처리 방식은 한 번 더 정리할 여지가 있습니다.
